## 3. Design Patterns & Architecture

### Three-Tier Plugin Architecture

```
┌───────────────────────────────────────────────┐
│   Commands (CLI Entry Points)                 │
│   /code-quality, /doctor, /delegate, etc.     │
└──────────────────┬────────────────────────────┘
                   │
┌──────────────────▼────────────────────────────┐
│   Skills (Domain Logic)                        │
│   prd, tdd-team, delegate-orchestrate, etc.   │
└──────────────────┬────────────────────────────┘
                   │
┌──────────────────▼────────────────────────────┐
│   Hooks & Infrastructure                       │
│   Security, fallback, review, config loading  │
└───────────────────────────────────────────────┘
```

### Key Architectural Patterns

#### 1. Multi-Agent Delegation with Auto-Routing

The system abstracts three heterogeneous AI agents behind a unified interface:

| Agent | Strength | Best For |
|-------|----------|----------|
| **Codex CLI** | Speed, low cost | Fast generation, scaffolding, simple edits |
| **Gemini CLI** | 1M+ context window | Large codebase review, full-context analysis |
| **Claude Subagent** | Best reasoning | Complex architecture, documentation, security |

Auto-routing via `delegate` skill classifies tasks and routes to the optimal agent. Fallback chain: Claude → Codex → Gemini.

#### 2. DAG-Based Task Orchestration

`delegate-orchestrate` decomposes complex requests into a dependency graph executed in waves:

```
Wave 1 (parallel):  task-1 [codex], task-2 [gemini]
Wave 2 (parallel):  task-3 [codex] (uses {{task-1.result}}), task-4 [codex] (uses {{task-1.result}})
Wave 3 (sequential): task-5 [claude] (uses {{task-3.result}} + {{task-4.result}})
```

Template placeholders (`{{task-N.result}}`) are replaced with actual results before downstream execution.

#### 3. TDD as Multi-Agent Workflow

The `tdd-team` skill orchestrates a 3-phase cycle per behavior:

| Phase | Agent | Responsibility | Constraint |
|-------|-------|---------------|------------|
| RED | Red Agent | Write failing test | Must compile but fail |
| GREEN | Green Agent | Simplest passing code | Minimum code only |
| REFACTOR | Refactor Agent | Clean up code | No behavior changes |

Features task decomposition, progress tracking (`[x]/[>]/[ ]` checkboxes), and user checkpoints between cycles.

#### 4. Hook-Based Security (Defense-in-Depth)

```
PreToolUse[Bash] → block-dangerous.sh
  ├── Block rm -rf outside PROJECT_DIR
  ├── Block indirect rm via bash -c, eval
  └── Normalize paths (resolve symlinks)

PostToolUse[Edit|Write] → post-edit-review.js
  ├── Advisory or blocking review mode
  ├── High-risk directory detection
  └── Debounced to avoid rapid reviews

Stop → on-stop.js
  ├── Detect rate limit / API errors
  ├── Auto-fallback for read-only tasks
  └── Notification for write tasks
```

#### 5. Configuration-Driven Agent Behavior

`.claude/orchestrator.local.md` (YAML frontmatter) controls:
- Agent enablement and binary paths
- Task-type → agent routing rules
- Fallback chain order
- Budget limits (max concurrent agents, delegation depth, cost caps)
- Post-edit review settings (mode, threshold, debounce)

Loaded by `config-loader.js` with deep-merge against sensible defaults.

### Extension Points

| Component | How to Add |
|-----------|-----------|
| **New Skill** | Create `skills/{name}/SKILL.md` with YAML frontmatter |
| **New Command** | Create `commands/{name}.md` with tool restrictions |
| **New Hook** | Add entry to `hooks/hooks.json` + script in `hooks/scripts/` |
| **New Agent Type** | Use `custom-agent-template` skill as scaffold |

## 7. Key Insights

`★ Insight ─────────────────────────────────────`
**1. Markdown-as-Code: Skills Are Programs Written in Natural Language**

The entire plugin system treats Markdown files as executable specifications. Each `SKILL.md` is not documentation — it's a program that Claude interprets and executes step-by-step. The YAML frontmatter serves as a function signature (name, version, triggers), while the Markdown body is the implementation. This is a fundamentally different paradigm from traditional plugin systems where code is compiled or interpreted by a runtime.

**Why it matters:** This approach makes skills accessible to non-programmers and enables rapid iteration (edit text, no build step), but shifts quality assurance from compilers/linters to evaluation frameworks like the TDD team benchmarking system.
`─────────────────────────────────────────────────`

`★ Insight ─────────────────────────────────────`
**2. DAG-Based Orchestration Solves the Multi-Agent Coordination Problem**

The `delegate-orchestrate` skill implements a wave-based execution model with template-based result injection (`{{task-N.result}}`). This is the same pattern used by build systems (Make, Gradle) and workflow engines (Airflow, GitHub Actions) — tasks declare dependencies, and the system automatically parallelizes independent work while respecting ordering constraints.

**Why it matters:** Without this pattern, multi-agent workflows would be either fully sequential (slow) or require manual coordination (error-prone). The DAG model gives automatic parallelization with correctness guarantees.
`─────────────────────────────────────────────────`

`★ Insight ─────────────────────────────────────`
**3. Hook-Based Security Is Defense-in-Depth for AI Agents**

The `block-dangerous.sh` hook intercepts Bash tool calls at the `PreToolUse` event, checking for `rm -rf` outside the project directory — including indirect execution via `bash -c` and `eval`. This is the same principle as OS-level system call interception (seccomp, AppArmor), applied at the AI tool-use layer.

**Why it matters:** Skills are user-contributed and may inadvertently trigger dangerous commands. By enforcing security at the hook layer (orthogonal to skill logic), the system provides safety guarantees without requiring every skill author to implement security checks.
`─────────────────────────────────────────────────`

`★ Insight ─────────────────────────────────────`
**4. The TDD Team Skill Demonstrates Stateful Multi-Turn Agent Orchestration**

Unlike simple delegation (send prompt, get result), the TDD team skill maintains state across multiple agent calls within a single user session: task list, current cycle, test results, progress checkboxes. Each RED/GREEN/REFACTOR phase is a separate Agent call, but they share context through the orchestrating skill's state management.

**Why it matters:** This pattern — a coordinator that maintains workflow state while delegating phases to specialized sub-agents — is a building block for complex agentic systems. It's analogous to a saga pattern in distributed systems, where a coordinator ensures multi-step transactions complete correctly.
`─────────────────────────────────────────────────`

`★ Insight ─────────────────────────────────────`
**5. Skill Duplication Reveals an Architectural Decision Point**

Both plugins define `codex-delegate` and `gemini-delegate` with different versions (v0.3.0 vs v0.1.0). This isn't accidental — it reflects a tension between **self-contained plugins** (each plugin works independently) and **DRY principles** (shared code shouldn't be duplicated). The toby-essentials versions are more mature, suggesting the orchestrator copies are legacy.

**Why it matters:** In plugin ecosystems, the "should we share or duplicate?" question is fundamental. Sharing creates coupling (plugins can't evolve independently); duplicating creates maintenance burden (bugs must be fixed in multiple places). The resolution is typically a "core" plugin that others depend on — a pattern this project hasn't yet adopted.
`─────────────────────────────────────────────────`

`★ Insight ─────────────────────────────────────`
**6. Benchmark-Driven Skill Development Is a Novel QA Pattern**

The `tdd-team-workspace` evaluation system compares old vs. new skill versions across multiple scenarios with explicit grading criteria (task decomposition, sequential Agent calls, progress tracking). This is closer to **ML model evaluation** (A/B testing with metrics) than traditional software testing (assert expected == actual).

**Why it matters:** When your "code" is a natural language specification interpreted by an LLM, traditional unit tests don't apply. Instead, you need scenario-based evaluations that measure whether the LLM follows the skill's instructions correctly. The 83.3% pass rate improvement from v0.2.0 to v0.3.0 demonstrates this approach catches real regressions.
`─────────────────────────────────────────────────`

`★ Insight ─────────────────────────────────────`
**7. Configuration Layering Enables Per-Project Customization Without Code Changes**

The `config-loader.js` implements a three-layer configuration model: hardcoded defaults → `.claude/orchestrator.local.md` overrides → deep merge. This mirrors patterns in Spring Boot (`application.yml` + profiles), Docker Compose (override files), and Kubernetes (ConfigMaps + defaults).

**Why it matters:** Users can customize agent routing, fallback chains, and budget limits per project without modifying plugin code. The deep-merge strategy means users only need to specify what differs from defaults, keeping configuration minimal.
`─────────────────────────────────────────────────`

`★ Insight ─────────────────────────────────────`
**8. Zero External Dependencies Is a Deliberate Architectural Choice**

Despite being a Node.js project, claude-orchestrator has **zero npm dependencies**. The YAML parser is hand-written (70 lines), the output parsers are custom, and the agent checker uses raw `child_process`. This eliminates supply chain risk, version conflicts, and `node_modules` bloat.

**Why it matters:** For a plugin that runs inside users' development environments, dependency minimalism reduces attack surface and installation friction. The trade-off is maintaining custom parsers (the regex-based YAML parsing in `post-edit-review.js` is already showing fragility), but for a project of this scale, the benefits outweigh the costs.
`─────────────────────────────────────────────────`
